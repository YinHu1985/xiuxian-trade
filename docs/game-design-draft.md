# 修仙商会网页游戏设计草案

## 一、项目定位

这是一款以“修仙世界中的商会经营”为主题的网页策略经营游戏。玩家将扮演一名商会主事人，驾驶飞舟穿梭于各个据点之间，在灾后重组的修仙世界中逐步建立贸易网络、扩张商会势力，并通过探索、经营与资源调度完成从一隅小镇到区域级商会的成长。

游戏的核心体验由以下三部分构成：

1. 驾驶飞舟在据点之间移动，进行低买高卖的贸易行为。
2. 在合适的据点设立商号和商路，构建可持续运转的经营网络。
3. 通过探索世界与收集稀有资源，为主角后续的修为成长预留发展空间。

当前阶段先聚焦贸易、探索与商会经营体系，主角修为成长仅保留接口与扩展位，暂不作为第一版重点玩法。

## 二、世界观与背景设定

多年前，一场覆盖广阔区域的天灾使原有文明秩序崩解，山河移位，灵脉紊乱，曾经连成一体的修仙世界被撕裂成大量相对独立的据点。如今，这片区域中散布着各种类型的节点，包括但不限于：

- 城镇
- 门派
- 遗迹
- 其他可扩展特殊据点

这些据点之间并非完全孤立，而是通过一条条危险、隐秘且信息不完整的道路彼此连接。玩家控制的商会将从初始城镇起步，在有限认知下探索周边区域，逐步掌握道路情报，发现新据点，建立商路，并最终形成覆盖整片区域的贸易网络。

## 三、核心玩法概览

玩家的主要游戏方式分为三条线：

### 1. 主角飞舟贸易

玩家直接控制主角的飞舟，在不同据点之间运输货物、倒卖商品、招募船员并规划航线。该部分是游戏前期最直接的资金来源，也是推动地图探索的主要方式。

### 2. 商会据点经营

玩家可在符合条件的据点设立商会分号。分号会根据当地人口、特产、周边道路条件和商路数量等因素，持续为商会提供回合收益。后续还可以通过扩建或修建子建筑，改变资源产出结构或强化某些经营能力。

### 3. 主角成长资源收集

主角本人存在修为成长的概念。玩家在贸易与探索过程中，可以同步收集未来升级所需的天材地宝。该系统在第一版中先作为长期目标预留，不深入展开具体养成规则。

## 四、单局目标与结局结构

### 1. 开拓许可背景

玩家并非无限期地经营这片区域，而是以“限期开拓者”的身份进入这片灾后地带。该区域在天灾后长期处于失序状态，外围大势力暂时无力彻底接管，于是向外部商会与个人势力发放临时开拓许可。

主角获得的正是一份有时限的开拓许可。在许可有效期内，主角及其商会可以较为优先地探索区域、经营据点、建立商路并争夺先发优势；一旦许可期结束，这片区域将进一步开放或被更大势力正式接手，主角也将失去当前局内所拥有的独占开拓窗口。

### 2. 单局时限

- 第一版按 `80` 回合设计
- 这 `80` 回合在叙事上对应一段固定期限的开拓经营期
- 回合结束后进入结局结算，而不是继续无限经营

### 3. 胜利结构

本质上，游戏采用“固定回合数结束后综合结算”的结构，但在表现层不直接强调抽象分数，而是以“开拓许可期满后的成果评定”来包装。

结局的核心判断逻辑是：主角是否在开拓许可期间建立了足够稳固的商业地位，并为商会在该区域后续的发展打下基础。

### 4. 结局评分维度

第一版结局评价先基于以下四个主要维度：

- 探索程度
- 商会规模
- 商路规模
- 资金数量

其中：

- 探索程度可参考地图开拓比例、已确认据点数量以及是否发现关键特殊据点
- 商会规模可参考已设分号数量、分号等级和经营布局
- 商路规模可参考已建立商路数量、关键节点控制情况和网络完整度
- 资金数量用于体现经营成果和资本积累水平

系统最终仍可计算总分，但结局文案优先强调“商会前景”“区域地位”“后续发展资格”等叙事化表达，而不是单纯展示一个分数数字。

### 5. 失败条件

第一版暂时只设置一种硬失败条件：破产。

建议采用如下判定方式：

- 在回合结算阶段，如果玩家持有的灵石不足以支付必要支出，则判定为破产
- 破产后立即结束本局，进入失败结局

即使进入失败结局，也仍会根据破产前已积累的探索、商会、商路和资金成果，给出不同层级的失败表述，体现“虽未守成，但仍留下若干影响”或“仓促败退，一无所成”等差异。

### 6. 后期终局目标

为了避免游戏后期仅剩下数值扩张和自动经营，第一版建议加入一个后期才会解锁的终局探索目标，为整局提供更强的叙事牵引。

建议方向如下：

- 玩家在完成一定商业积累后，可以启动一项耗资巨大的区域工程
- 该工程的典型方案为修复一座灾前遗留下来的大型传送阵
- 传送阵修复后，可通往更深区域、关键遗迹，或揭示天灾相关的终极秘密

该设计的意义在于：

- 前期通过跑商与开图完成原始积累
- 中期通过商号与商路建立稳定经营能力
- 后期再用商业成果反向推动一次高规格探索行动

这样可以让“经营”与“探索”在终局阶段重新汇合，强化沉浸感与成长感。

## 五、回合与行动规则

游戏采用回合制推进，但并非所有操作都会消耗回合。

### 1. 不消耗回合的常规操作

- 在据点内买卖商品
- 查看信息
- 普通经营管理
- 招募等基础交互

### 2. 消耗回合的特定操作

- 飞舟移动
- 修整
- 探索
- 其他明确标记为“推进时间”的行动

### 3. 单回合行动框架

第一版建议采用清晰的“一回合一个耗时决策”结构：

- 玩家在当前据点内可执行任意数量的不耗时操作
- 在完成准备后，玩家本回合只能选择 `1` 个耗时动作
- 耗时动作执行后，回合推进，世界进入下一轮结算

这样可以保证每回合的关键抉择足够明确，便于玩家权衡“继续准备”还是“立即推进时间”。

### 4. 标准回合流程

建议的标准回合流程如下：

1. 回合开始，结算上一轮世界变化与持续收益
2. 玩家在当前据点进行任意不耗时操作，例如买卖、查看情报、管理商号等
3. 玩家选择一个耗时动作，例如移动、修整、探索或特定经营行为
4. 执行耗时动作后，推进回合并触发下一轮世界刷新与结算

这套流程的好处在于：

- 据点内操作足够自由
- 时间推进足够清晰
- 玩家每回合都必须作出一个明确的战略选择

### 5. 飞舟移动规则

- 主角拥有一艘飞舟作为主要行动载具。
- 每回合最多移动 `n` 格路径。
- 初始移动力为 `1`。
- 后续可通过游戏进展、系统升级或载具强化提高移动力上限。

这种设计使玩家可以在据点内高频进行策略操作，同时将“时间”更多绑定在旅行、探索和恢复等关键抉择上。

## 六、大地图与随机生成

### 1. 地图生成目标

每一局游戏的大地图都应随机生成，从而保证重玩价值。地图中会生成一定数量的不同类型据点，并为这些据点建立路径网络。

### 2. 据点生成要求

- 随机生成多个据点
- 据点拥有类型差异，例如城镇、门派、遗迹等
- 不同类型据点可决定其功能、产出和经营价值

### 3. 道路生成要求

路径生成需要满足以下原则：

- 所有节点必须整体联通，不能出现完全孤立的据点
- 路径不要求总是连接最近节点
- 一个节点可以连接多个节点，也可以只有较少连接
- 整体连接数量不宜过多，以保证探索价值、路径选择和商路布局的游戏性

目标效果接近“星图网络式”的据点连接结构：整体可达，但局部稀疏，便于形成关隘、支路、枢纽和高价值通道。

### 4. 开局可选参数

地图生成的关键参数建议在开局时作为可选项暴露给玩家，用于控制单局长度、探索密度和经营压力。第一版可以先支持以下参数：

- 据点总数
- 据点类型比例
- 起始区域已知范围
- 道路连接密度
- 枢纽点、深区据点与死胡同宝点的分布倾向

这些参数既可以服务不同偏好的玩家，也便于后续测试平衡时快速观察体验差异。

### 5. 默认推荐参数

在第一版中，建议先使用一组较稳妥的默认值，再根据试玩结果持续调整：

- 据点总数：`18-24`
- 据点类型比例：城镇 `50%`，门派 `20%`，遗迹 `20%`，特殊据点 `10%`
- 起始缓冲区：出生点外加周边 `2-3` 个据点，以及其对应的部分道路信息
- 平均连接度：每个据点平均 `2-3` 条连接，少量枢纽点可达到 `4` 条
- 地图特征：默认存在少量枢纽点、少量深区高价值据点、少量死胡同型宝点

这组参数的目标是让地图既有可探索空间，又不至于在 80 回合内过于松散或过于拥挤。

### 6. 地图结构意图

默认地图结构建议具备以下特征：

- 起始区域相对安全，便于玩家建立初步贸易循环
- 中段区域逐渐出现枢纽点，方便形成商路争夺和路线抉择
- 深区区域保留更高价值的非城镇据点，鼓励玩家向外推进
- 局部存在死胡同型特殊节点，用于承载稀有资源或一次性高价值探索目标

这样可以让玩家在不同阶段自然经历“起步”“扩张”“冲击深区”的节奏变化。

## 七、战争迷雾与探索信息分级

游戏开局时，玩家并不掌握完整地图信息。

### 1. 初始可见范围

- 开局仅显示出生点
- 同时显示出生点周边约两到三个据点及其部分道路信息
- 其余据点和道路默认隐藏

### 2. 据点探索度分级

据点信息分为三级：

1. 未知：完全不存在于玩家视野中
2. 传闻：玩家听说过该据点的存在，地图上可显示，但信息不完整
3. 确认：玩家已确认该据点真实存在，并掌握稳定信息

### 3. 路径探索度分级

路径信息分为四级：

1. 未知：玩家不知道该道路存在
2. 传闻：玩家听说存在这条道路，但尚未完全确认
3. 确认：玩家已确认该道路存在并可正常利用
4. 商路：玩家已对该路径进行进一步投入，使其成为正式商路

### 4. 信息获取逻辑

- 玩家到达某个据点后，可以通过“打听”等城镇行为获取相连据点的信息
- 例如在 A 据点打听到 B，则 B 可从“未知”变为“传闻”
- 同时，A-B 之间的道路也会获得相应传闻信息
- 当玩家确认 B 的存在时，可视为同时确认 A-B 道路存在
- 但若 B 还与 C 相连，则 C-B 是否存在仍需单独确认，不会因为确认 B 而自动揭示

### 5. 状态独立原则

据点状态与路径状态相互关联，但不是完全绑定：

- 确认据点，通常可同步确认通往该据点的特定道路
- 其他与该据点相关的道路，仍需逐条探索或确认
- 已确认道路后，还可以继续投入资源，将其升级为商路

这一机制可以让玩家逐步“织出”自己的认知地图，而不是一次性点亮整张世界图。

### 6. 打听规则

第一版中，玩家在城镇类据点可通过酒馆“请喝酒”的方式打听消息。

- 每次打听消耗少量灵石
- 打听本身不消耗回合
- 每次打听有一定概率获得当前据点周边的有效道路或据点传闻
- 当次未获得有效信息时，可用无实际价值的闲谈、传闻或地方轶事填充反馈，以增强沉浸感

在体验设计上，打听不应成为高门槛行为。第一版可将其设计为“成本很低、可多次尝试”的轻量交互，使玩家在抵达新据点后，通过多按几次就能逐步摸清周边局势，而不会因为单次随机失败而卡住节奏。

### 7. 打听范围

第一版建议将打听结果限制在“当前据点直接相邻的内容”上：

- 优先揭示与当前据点直接相连的据点传闻
- 优先揭示当前据点出发的相邻道路传闻
- 默认不直接跨两层揭示更远区域

这样可以让地图认知以据点为中心逐步外扩，保持探索感和空间层次感。

### 8. 探索与确认规则

“传闻”到“确认”的过程需要通过耗时动作完成，而不是只靠打听直接点亮。

- 主角执行探索或确认道路时，需要先移动到目标据点
- 探索或确认会消耗回合
- 第一版中，探索的主要功能是把已有传闻转化为确认信息
- 第一版不必为探索额外叠加过多复杂随机事件，以免节奏过散
- 未确认的据点也允许玩家提前得知其潜在产出，用于提前规划后续跑商路线
- 当玩家执行“探索道路并确认目标据点”的耗时动作后，回合结束时主角会顺便移动到目标据点

该设计保证了信息获取分成两个阶段：

- 打听负责低成本获知线索
- 探索负责投入时间完成实地确认

这样既保留了情报收集的轻松感，也保留了亲自跑图的行动价值。

### 9. 主角与供奉的分工

为了避免在 80 回合内仅靠主角一人无法有效处理中后期的大量探索和建设任务，第一版建议加入“供奉”系统，作为有限度的额外执行力来源。

- 主角拥有飞舟，需要实际移动到目标据点后才能执行耗时任务
- 商会发展到一定程度后，可雇佣少量供奉
- 供奉可以直接前往任意已确认据点执行部分耗时任务
- 供奉适合承担探索、确认、升级商路等标准化事务

该设计的核心目的不是彻底解放玩家，而是在中后期为玩家补充有限的行动并行能力。

### 10. 供奉系统的边界

供奉系统应当是辅助，而不是取代主角。

- 供奉数量应保持较低上限，第一版建议控制在 `2-3` 名
- 供奉只能前往已确认据点，不能直接代替主角开辟完全未知区域
- 供奉只执行部分标准化耗时任务，不承担全部经营与剧情功能

这样既能缓解后期回合数紧张的问题，又不会让每回合需要处理的事务膨胀失控。

### 11. 探索完成度预期

单局结束时，地图仍有部分区域未被确认是完全可以接受的。

- 游戏目标不是要求玩家在一局内彻底清空整张地图
- 玩家更需要在有限许可期内判断哪些区域值得优先投入
- 这也使不同局之间保留了更多不确定性和重玩空间

因此，探索系统应鼓励“有选择地深入”，而不是逼迫玩家追求全图完美确认。

## 八、据点类型与功能定位

当前可先将据点理解为若干具备差异化产出和功能的节点，后续再细分。

### 1. 据点核心字段

第一版不做复杂城镇建设模拟，而是将大多数据点的经营相关数值先收敛到两个核心字段：

- 繁荣度
- 修为层次

这两个字段承担的职责如下：

- 繁荣度主要决定商业活跃程度、商品整体刷新数量和库存上限
- 修为层次主要决定高阶商品是否解锁，以及高阶商品的刷新数量和库存上限

这种简化可以让第一版更聚焦于贸易与开拓，而不是陷入过度细碎的城市经营参数。

### 2. 据点类型与初始倾向

不同类型的据点可以共享这两个核心字段，但拥有不同的初始倾向。

- 城镇通常具备较高的初始繁荣度，但修为层次相对一般
- 门派、遗迹等非城镇据点通常繁荣度较低，但修为层次更高
- 特殊据点可根据主题决定偏向，例如矿区偏产出，秘境入口偏高修为低繁荣

这使不同类型据点在同一套基础框架下仍能呈现出明显差异。

### 3. 城镇

- 玩家初期发展的核心据点
- 适合进行基础贸易、补给、招募和设立商号
- 更适合作为商业中枢和商号经营节点
- 可通过商会投资提升繁荣度与修为层次

### 4. 城镇开发原则

- 商会原则上可以将任意城镇逐步开发到较高水平，甚至推向顶级发展状态
- 低基础城镇若强行投资，往往回本效率较差，但不阻止玩家出于沉浸感或个人偏好进行培养
- 若某个低基础城镇本身位于关键枢纽位置，则其区位优势可以部分抵消基础薄弱带来的劣势

这意味着“把一座偏僻小城硬生生经营成大埠”可以成为一种可行但未必最优的玩法乐趣。

### 5. 门派

- 可作为高端修仙资源的重要来源地
- 适合产出高阶丹药、功法相关资源或特殊订单
- 也可在后续扩展为声望、关系系统的重要节点

### 6. 遗迹

- 可作为稀有原料和特殊探索收益的主要来源
- 更偏向探索型据点
- 可承载一次性收益、周期性开发或高风险高回报玩法

### 7. 其他特殊据点

为后续扩展保留，例如：

- 灵脉矿区
- 宗门坊市
- 秘境入口
- 渡口要塞

## 九、贸易系统设计

### 1. 核心货币

游戏的主资源为灵石，承担“资金”功能，是贸易、建设、升级和投入商路的基础通货。

### 2. 商品维度

商品由三个维度交叉组合生成：

- 类别
- 等级
- 属性

当前设定如下：

#### 类别

1. 药材
2. 金石
3. 毛皮
4. 灵液
5. 符篆
6. 丹药
7. 装备

#### 等级

1. 炼气
2. 筑基
3. 金丹
4. 元婴

#### 属性

1. 金
2. 木
3. 水
4. 火
5. 土

按上述组合，基础商品池总量为：

`7 * 4 * 5 = 140`

### 3. 装备命名规则

装备类商品在不同等级下采用特殊命名：

- 炼气：兵器
- 筑基：法器
- 金丹：灵器
- 元婴：法宝

这意味着装备在数据结构上仍属于统一的“装备类”，但在展示层和文本命名上可以根据等级映射为不同称呼。

### 4. 据点产出限制

类似传统航海贸易游戏，不同据点只会产出特定种类、特定属性的商品，而不是全品类生产。

例如：

- 某城镇主要产出火属性药材
- 某矿区主要产出金属性金石
- 某遗迹主要产出高阶特殊原料
- 某门派主要产出高端丹药

在产出信息层面，第一版允许玩家在据点尚未完全确认前，就先获知该据点的潜在产出方向。这样玩家可以在探索过程中提前规划“顺路跑商”，而不是必须等所有信息完全确认后才能开始形成贸易判断。

### 5. 繁荣度与高阶解锁

- 繁荣度主要影响整体商品刷新数量与库存上限
- 修为层次主要影响高阶商品的解锁、刷新数量与库存上限
- 当某据点理论上可产出某种高阶商品，但当前繁荣度或修为层次不足时，实际刷新量或库存上限可能在取整后变为 `0`
- 这意味着玩家可能会提前知道某据点具备高端商品潜力，但必须经过开发后才能真正稳定产出
- 高端商品不应全部由普通城镇提供
- 部分顶级商品应限定在非城镇据点产出，例如门派、遗迹等

这样可以确保玩家不会只在城镇之间循环贸易，而必须进入更深区域、接触更多特殊据点，才能打通高价值贸易链。

### 6. 库存刷新

第一版先采用较为简单直接的库存模型：

- 每回合，每种商品增加一定数量库存
- 每种商品都存在库存上限
- 刷新量与上限最终都取整数，便于展示和理解
- 繁荣度主要放大整体刷新量与整体库存上限
- 修为层次主要决定高阶商品是否有资格参与刷新，以及其刷新量与上限的高低

该模型足以支持第一版的基础跑商体验，同时保留了后续追加更复杂供需公式的空间。

## 十、价格波动与市场影响

玩家的买卖行为会影响市场价格，且影响不是只针对单一商品，而是作用在多个维度上。

### 1. 基础价格规则

第一版建议先采用足够清晰、玩家容易心算的价格模型。

- 每种商品拥有一个基础价格
- 只要某据点会自然刷新某商品，就视为该商品的原产地之一
- 在原产地，玩家可以按基础价格的 `50%` 买入该商品
- 在非原产地，玩家默认按基础价格卖出该商品
- 为了降低误操作惩罚，玩家在原产地也允许按基础价格的 `50%` 卖出

例如，某商品基础价格为 `100`：

- 在原产地可以买入价 `50`
- 在非原产地默认卖出价 `100`
- 在原产地误卖时，也至少能按 `50` 售出

### 2. 系数叠加方式

在基础价格和原产地半价规则之上，再叠加两个本地市场系数：

- 类别系数，取值区间建议为 `[0.5, 1.5]`
- 属性系数，取值区间建议为 `[0.5, 1.5]`

第一版建议采用如下规则：

- 类别系数和属性系数同时影响买价与卖价
- 这两套系数按据点分别维护，属于据点本地市场状态，而不是全图共享状态
- 实际价格可按 `基础价格 * 类别系数 * 属性系数` 计算

例如，某商品基础价格为 `100`，若当前据点的类别系数为 `1.2`，属性系数为 `1.1`，则该商品在该地的实际价格为 `132`。

### 3. 市场波动规则

- 每买入或卖出 `1` 份商品，都会对对应类别系数和属性系数各影响 `0.01`
- 大量采购会推高对应类别与属性商品的本地价格
- 大量出售会压低对应类别与属性商品的本地价格
- 每回合结束后，所有市场系数都会向 `1` 回归 `0.05`

这样一来，价格既会受到玩家即时交易行为影响，又会随时间逐步恢复，避免单地市场长期锁死在极端状态。

### 4. 示例

如果玩家在某座城内大量采购“药材”，则该城的药材类别系数会上升；如果同时采购的是“水属性药材”，则该城的水属性系数也会上升。若某件商品同时属于“药材”和“水属性”，则它会同时受到两个维度的价格联动影响。

### 5. 设计意义

这一机制可以避免价格系统过于静态，使玩家在规划贸易时需要考虑：

- 当前采购行为对后续交易的影响
- 单一爆买是否会削弱利润空间
- 类别维度与属性维度叠加后的市场联动
- 各地本地市场差异所带来的空间性机会

## 十一、飞舟与货仓系统

### 1. 飞舟定位

飞舟是玩家在世界地图上的核心行动单位，负责移动、运输货物和承载探索行动。

### 2. 货仓机制

飞舟拥有有限货仓，建议采用类似“大航海时代 4”的舱位式设计。

- 每个货仓格可以装载一定数量货物
- 货仓容量限制直接决定单次贸易规模
- 后续可通过升级飞舟、改装模块或解锁更大载具提高载货量

### 3. 经营意义

货仓限制会迫使玩家在“高利润低体积”“基础货物稳定周转”“探索补给占位”等需求之间作出取舍。

## 十二、商号经营系统

### 1. 设立条件

主角在经过合适的城镇据点时，可以在该据点开设商会分号。

### 2. 分号收益来源

第一版中，商号本身提供被动收益，收益核心取决于据点当前能够支配的“产品种类池”。

可理解为：

- 某据点可支配的产品种类越多，商号的经营覆盖面越广
- 产品种类池越丰富，商号每回合带来的被动收益越高
- 建筑、贸易连接和周边收集行为，本质上都是通过扩充产品种类池来提升商号收益

为降低第一版复杂度，产品池的计数先按“有无”判断，而不按具体数量叠加收益。

### 3. 产品池规则

第一版建议采用如下简化规则：

- 某种产品只要在该据点可被视为“存在”，就计入产品种类池
- 同类产品不因数量更多而重复累加商号收益
- 建筑产出、周边并入产出、贸易连接带来的外来产出，都可以为商号收益提供支持
- 但不同来源的产品要区分用途，避免在系统之间无限转发

这种做法可以让经营层更聚焦在“网络覆盖面”和“产业链结构”上，而不是陷入过早的数量微操。

### 4. 子建筑类型

商号可以建设若干功能明确的子建筑。第一版先建议保留以下三类：

#### 4.1 集散行

`集散行` 用于吸纳周边资源。

- 可将周边 `n` 格内据点的物产并入本据点的产品池
- 第一版默认 `n = 1`
- 被并入的周边物产在产量或上限上可按一定系数折算
- 对第一版的商号收益而言，重点仍是“是否纳入产品池”，而不是具体数量精算

#### 4.2 作坊类建筑

作坊负责把已有原料来源转化为更高价值的产品种类。

第一版可先支持以下方向：

- `丹房`：需要药材来源，增加丹药类产出
- `器坊`：需要金石与毛皮来源，增加装备类产出
- `符坊`：需要毛皮与灵液来源，增加符篆类产出

为简化实现，第一版中作坊只判断“是否具备原料来源”，暂不精细计算原料消耗数量。只要原料来源存在，对应作坊即可生效。

#### 4.3 拍卖行

`拍卖行` 不负责新增产品，而是直接对商号收益提供比例加成。

- 更适合建在已经具备较丰富产品池的成熟据点
- 可作为中后期强化核心商业节点的手段

### 5. 开发方向与沉浸感

商号系统允许玩家走两种不同方向：

- 在天然条件优越的大城扩张，追求更高效率
- 在区位优秀但基础薄弱的小城重金投入，追求“亲手养成一座商业枢纽”的沉浸感

即使后者未必是纯数值上的最优解，也应当作为被允许且有趣的玩法存在。

## 十三、商路系统

### 1. 概念区分

为了避免概念混淆，第一版建议将两个概念区分开：

- `贸易连接`：由商会主动建立的、连接两个已确认据点的经营关系
- `道路熟悉度`：某条已确认道路在反复使用后逐步降低贸易成本的状态

玩家习惯上仍可将其统称为“商路”，但在设计文档和数据结构中最好分层处理。

### 2. 建立条件

玩家可在两个据点之间建立一条贸易连接，前提至少包括：

- 两个据点都已经探索确认
- 两个据点之间存在一条可通行的已确认路径
- 玩家当前可用的贸易连接名额尚未用尽

贸易连接总数需要设置上限，并可随着主角或商会发展逐步解锁更多名额，类似供奉名额的扩张逻辑。

### 3. 贸易连接的作用

贸易连接本身不直接产出灵石，反而是持续支出项。它的核心作用，是让连接首尾的两个据点同时获得对方的产出信息与经营价值。

具体表现为：

- 连接两端据点会互相获得对方的产品种类
- 这些新增产品可计入本据点的产品池，从而提高商号收益
- 这些新增产品也可以作为本地作坊的原料来源
- 但贸易连接带来的外来产品不会继续转发到其他贸易连接中
- 同样，`集散行` 并入的周边产出也不应被继续多级转发

这样可以避免经营网络出现无限复制和指数膨胀。

### 4. 主角买卖与经营产出的边界

为了保持手动跑商与后台经营的分工清晰，第一版明确区分“可经营利用的产品”与“可被主角直接采购的大宗商品”。

- 主角买卖的大宗商品，只能采购当地原产商品
- 建筑新增产出可以用于提高商号收益，也可以作为作坊原料
- 贸易连接带来的外来产品只参与经营层，不进入主角可直接采购池

这意味着商号与贸易连接是在做“经营网络扩张”，而主角本人仍主要围绕本地原产商品进行手动跑商。

### 5. 道路熟悉度

第一版不再单独设计“商路探索”阶段，而是改为让道路在使用中自然成长。

- 只要道路已确认，玩家就可以立刻沿该道路进行贸易
- 初期通过该道路时，贸易成本较高
- 随着该道路被持续使用、经过更多回合后，道路熟悉度会逐步提升
- 道路熟悉度提升后，可降低通过该道路的贸易成本，从而提高跑商利润

为简化实现，第一版建议将道路熟悉度设计为一个低复杂度等级值，例如 `0-3` 或 `0-5`，并随使用次数或累计经过回合自动成长。

### 6. 系统意义

这套设计使商号、贸易连接与主角跑商形成明确分工：

- 商号负责把据点经营成更大的商业节点
- 建筑负责扩展产品池与加工能力
- 贸易连接负责把多个节点织成经营网络
- 主角跑商负责处理本地原产商品的即时套利

这样既能保留手动贸易的操作乐趣，也能让中后期经营系统真正体现“商会版图”的成长感。

## 十四、主角成长系统预留

主角存在修为成长设定，玩家在游戏过程中需要收集升级所需的天材地宝。

不过，当前版本暂不深入实现该系统，仅保留以下设计方向：

- 主角成长资源可通过贸易、探索、特殊据点或任务获得
- 不同境界可能对应不同等级与属性的材料需求
- 后续可与装备、门派、秘境和高端商品系统联动

第一版建议只保留数据接口和资源归类，不强行展开具体数值与成长流程。

## 十五、数据结构与可扩展性要求

在设计底层数据结构时，需要明确为后续扩展预留空间，而不是将当前版本写死。

### 1. 商品系统需可扩展

需要支持未来追加：

- 新境界等级
- 新商品类别
- 新属性体系
- 特殊商品标签

### 2. 据点系统需可扩展

需要支持未来追加：

- 新据点类型
- 新建筑类型
- 特殊地块效果
- 区域级规则或势力归属

### 3. 地图系统需可扩展

需要支持未来追加：

- 更复杂的道路类型
- 特殊通行限制
- 区域事件
- 多层地图或特殊空间结构

### 4. 角色与商会系统需可扩展

需要支持未来追加：

- 主角修为成长
- 船员系统细化
- 商会科技或制度升级
- 势力关系、任务和事件链

### 5. 配置系统需可扩展

除内容扩展外，第一版还应当明确支持“数值配置可调”，避免将关键平衡参数写死在代码逻辑中。

需要支持未来通过配置表追加或调整：

- 地图规模、据点比例、连接密度等开局参数
- 商品基础价格、原产地折扣、库存刷新速度、库存上限等经济参数
- 市场系数波动幅度、回归速度、价格上下限等价格参数
- 供奉名额、贸易连接名额、移动力成长等进程参数
- 事件触发概率、终局目标门槛、结局评分权重等流程参数

推荐做法是将这类参数统一放入独立配置结构中，由“默认配置 + 开局覆盖项”共同决定单局实际规则。

### 6. 存档结构需可扩展

游戏状态同样需要按可扩展思路设计，避免未来新增字段时破坏旧档。

建议原则如下：

- 存档采用结构化数据，优先使用清晰的对象字段而不是位置数组
- 存档中包含 `version` 字段，用于后续兼容与迁移
- 配置数据、地图状态、玩家状态、据点状态、道路状态、商号状态、贸易连接状态应分模块存储
- 新系统追加时应尽量以新增字段方式扩展，而不是修改旧字段语义

这样后续无论是本地读档还是 JSON 导入导出，都更容易维护兼容性。

## 十六、开局可调参数与配置体系

### 1. 设计原则

所有明显影响游戏体验和平衡性的关键系数，第一版都应尽量做成可调参数，并允许玩家在开始游戏前进行设置。

这样做的意义包括：

- 方便快速测试不同节奏下的可玩性
- 允许玩家选择更偏探索、经营或紧凑节奏的开局
- 为后续难度模式、自定义模式和版本平衡预留空间

第一版不必做过于复杂的高级编辑器，但至少应提供一套“预设值 + 可手调参数”的开局配置界面。

### 2. 参数分类

建议将开局可调参数分为以下几类：

#### 2.1 地图参数

- 据点总数
- 据点类型比例
- 起始已知范围
- 平均连接度
- 枢纽点、深区节点、死胡同宝点的分布倾向

#### 2.2 回合与进程参数

- 单局总回合数
- 初始移动力
- 供奉初始上限与成长节奏
- 贸易连接上限与成长节奏
- 终局目标解锁门槛

#### 2.3 经济参数

- 商品基础价格
- 原产地价格折扣
- 库存每回合刷新量
- 库存上限系数
- 商号基础收益系数
- 拍卖行收益加成比例

#### 2.4 市场参数

- 类别系数上下限
- 属性系数上下限
- 单位买卖对价格系数的影响值
- 每回合价格系数回归速度
- 道路贸易成本基础值
- 道路熟悉度成长速度

#### 2.5 探索与情报参数

- 打听单次花费
- 打听获得有效信息的概率
- 集散行收集范围
- 周边物产并入折算系数
- 探索或确认动作的耗时规则

### 3. 默认值与开局覆盖

推荐采用“默认规则集 + 开局覆盖项”的结构：

- 系统内置一组平衡后的默认值，作为标准模式
- 玩家在开始新游戏时，可以按选项覆盖其中部分参数
- 生成的新局需要把最终生效的参数快照写入存档

这样做可以保证：

- 后续版本修改默认值时，不会影响旧档规则
- 导出存档时，其他人也能完整复现这局游戏的规则环境

### 4. 预设模式建议

除了手动调参外，第一版还可以提供若干预设模式，方便快速开局：

- 标准模式：使用推荐默认值
- 探索模式：更大地图、更高情报收益、更慢经济压力
- 经营模式：更高商号与贸易连接权重、更强调据点开发
- 紧凑模式：更少回合、更快价格波动、更高决策压力

这些预设本质上只是对同一套参数配置的不同打包方式。

## 十七、存档、读档与 JSON 导入导出

### 1. 本地存档目标

第一版应提供基础存档与读档能力，至少满足以下需求：

- 可在本地保存当前游戏进度
- 可从本地继续读取已有存档
- 可同时保留多个存档位
- 可显示基础存档信息，例如日期、回合数、灵石数量、地图进度

### 2. 存档内容范围

每个存档至少需要保存以下模块：

- 当前生效的开局配置参数
- 地图生成结果与随机种子
- 据点状态、道路状态与道路熟悉度
- 玩家位置、飞舟状态、货仓内容与灵石数量
- 商品库存、市场系数与刷新状态
- 商号、建筑、供奉与贸易连接状态
- 当前回合数、结局目标进度与其他流程状态

其中“开局配置参数”和“随机种子”尤其重要，它们决定了读档后是否能准确还原原局面。

### 3. JSON 导出与导入

第一版建议允许将存档导出为 JSON，并支持从 JSON 导入。

导出 JSON 的价值包括：

- 方便玩家备份
- 方便玩家分享特定局面
- 方便开发阶段排查 bug 与复现实验局
- 方便未来做社区挑战种子或规则复现

导入 JSON 时，建议进行基础校验：

- 检查 `version`
- 检查必要字段是否存在
- 检查数值范围是否合理
- 对缺失但可兼容的字段填入默认值

### 4. 存档格式建议

JSON 存档建议采用模块化结构，例如：

- `meta`：版本、时间、描述信息
- `config`：本局实际生效配置
- `world`：地图、据点、道路、随机种子
- `player`：主角、飞舟、货仓、灵石
- `economy`：库存、市场系数、价格状态
- `guild`：商号、建筑、供奉、贸易连接
- `progress`：回合、终局目标、结局标记

这样后续扩展新系统时，只需增加新模块或新字段，而不必重写整个存档协议。

### 5. 兼容性原则

为了兼顾未来扩展，第一版就应提前约定：

- 导出的 JSON 尽量使用稳定字段名
- 新版本读取旧档时，优先向前兼容
- 若出现重大不兼容变更，应通过 `version` 和迁移逻辑处理
- 配置快照必须写入存档，不能只依赖当前全局默认值

## 十八、第一版最小可玩原型范围

### 1. 第一版必须实现

为了保证能尽快得到可验证的可玩原型，第一版建议必须实现以下内容：

1. 新游戏开局与基础参数设置
2. 随机地图生成与据点连线
3. 据点/道路的未知、传闻、确认状态
4. 酒馆打听、探索确认与道路熟悉度成长
5. 主角飞舟移动、回合推进与货仓买卖
6. 基础商品系统、原产地半价、库存刷新与本地市场系数
7. 商号建立、产品池计算与核心子建筑
8. 贸易连接建立、产品共享与经营收益结算
9. 供奉的基础派遣能力
10. 80 回合结局、破产失败与多维结局结算
11. 本地存档、读档与 JSON 导入导出

### 2. 第一版可保留为简化实现

以下内容建议在第一版先采用简化版，而不是追求一次做满：

- 作坊只判定原料来源，不做复杂消耗
- 商号收益按产品种类池计算，不先做复杂数量公式
- 贸易连接不做复杂物流模拟，只做产品共享与支出
- 道路熟悉度采用低复杂度等级成长
- 供奉数量保持较少，功能只覆盖标准化任务

### 3. 第一版暂缓实现

以下内容继续保留为后续扩展项：

- 主角完整修为养成
- 深度剧情与任务链
- 势力外交与复杂关系
- 战斗系统
- 更复杂的事件系统
- 高级城市建设模拟

### 4. 第一版开发原则

第一版的目标不是把所有设想一次做完，而是先验证以下四件事：

- 随机开图是否有乐趣
- 跑商与价格波动是否形成稳定决策
- 商号与贸易连接是否能构成有意义的经营网络
- 80 回合的整体节奏是否顺畅

只要这四点成立，后续就有非常稳固的迭代基础。
